# Step 01.5 — Pre-review simplify pass

Runs the host's quality-only `/simplify` skill over the change set
**before any internal review** (step 02 adversarial review, step 04g
`/code-review`). Rationale: `/simplify` is reuse/clarity/efficiency
cleanup only — it does **not** hunt for bugs. Running it first means the
review surfaces critique already-clean code, so the loop churns less on
"this is overly complex" findings. This mirrors the user's global
working rule "run /simplify before the verify gate — it edits the tree."

Decimal name (`01.5`) follows the `04.5-local-verify.md` precedent for a
step inserted between two existing numbered steps; do not renumber
`02`/`03`.

## Host gating

`/simplify` is a claude-code skill. Like step 04g's `review` mapping,
this step is host-specific:

| `context.host_platform` | Skill name | Invocation | Runs? |
|---|---|---|---|
| `claude-code` | `simplify` | `Skill(skill="simplify")` | Yes |
| `codex` | (not mapped) | Skip; log `simplify_invoked {skipped: true, reason: "host-unmapped", host: <host_platform>}` | No |
| `gemini` | (not mapped) | Skip; log `simplify_invoked {skipped: true, reason: "host-unmapped", host: <host_platform>}` | No |
| `other` | (none) | Skip; log `simplify_invoked {skipped: true, reason: "host-unmapped", host: <host_platform>}` | No |

When skipped, **set `context.simplify_outcome = "skipped"`** and proceed
directly to step 02 with `context.head_sha` unchanged. `Skill` MUST be
present in the orchestrator `allowed-tools` (it is also required by step
04g).

> Each terminal branch of this step sets `context.simplify_outcome`
> explicitly (one of `skipped` / `no-changes` / `committed` / `folded` /
> `rolled-back`) alongside its log event — never leave it null, or the
> step-11 report and the `committed`-outcome invariant below have nothing
> to read.

## Skip conditions

Skip the step (log `simplify_invoked {skipped: true, reason: ...,
host: <host_platform>}` — always include the `host` field per
log-format.md's declared payload) and proceed to step 02 unchanged when
any apply:

1. `context.host_platform != "claude-code"` (see table; `reason:
   "host-unmapped"`).
2. The review range is empty — there is nothing to simplify (`reason:
   "no-changed-code"`). `context.base_sha` is not exported as a shell
   var by platform detection, so resolve it here:
   ```bash
   BASE_SHA="$(git merge-base "$BASE" HEAD)"   # $BASE = context.base
   if git diff --quiet "$BASE_SHA"..HEAD && [ -z "$(git status --porcelain)" ]; then
     : # range empty → skip the step, proceed to step 02 unchanged
   fi
   ```

## Pre-state capture (attribution)

`/simplify` edits the working tree. To commit its delta **separately**
(the chosen model) we must distinguish simplify's edits from any
pre-existing uncommitted implementation work.

Before invoking the skill, snapshot the already-dirty set:

```bash
# Paths that are dirty BEFORE simplify runs (committer hasn't staged
# them yet). Empty in the dominant case: implementation work is
# committed and the tree is clean. `cut -c4-` takes the path field
# (porcelain reserves columns 1-3 for the XY status + separator),
# correct for paths containing spaces; awk '{print $2}' would split them.
PRE_DIRTY=$(git status --porcelain | cut -c4- | sort -u)
```

Path selection keys on the **related** subset of `PRE_DIRTY` (per step
01's relatedness heuristic) — unrelated dirty files never force the fold
path and are never staged by this step.

- **Committed case** — no *related* file is in `PRE_DIRTY` (the
  implementation work is committed; the tree is clean apart from
  possibly-unrelated noise). Simplify's edits to committed files are
  cleanly separable → **separate-commit path**, staging only
  `SIMPLIFY_FILES` (the `comm -13` set: newly-dirtied by simplify).
- **Overlap case** — at least one *related* file is in `PRE_DIRTY`.
  Simplify may edit those same files in-place, intermingling its delta
  with un-committed implementation work that cannot be split per-hunk
  safely → **fold path**.

## Invoke

Unless `--dry-run` (see "Dry-run" below):

```
Skill(skill="simplify")
```

After it returns, recompute the dirty set and derive simplify's touched
files:

```bash
POST_DIRTY=$(git status --porcelain | cut -c4- | sort -u)
# Files newly dirty, or dirty-and-also-pre-dirty, after simplify:
SIMPLIFY_FILES=$(comm -13 <(printf '%s\n' "$PRE_DIRTY") <(printf '%s\n' "$POST_DIRTY"))
OVERLAP_FILES=$(comm -12 <(printf '%s\n' "$PRE_DIRTY") <(printf '%s\n' "$POST_DIRTY"))
```

If `SIMPLIFY_FILES` and `OVERLAP_FILES` are both empty, `/simplify` made
no edits → set `context.simplify_outcome = "no-changes"`, log
`simplify_no_changes`, proceed to step 02 with `context.head_sha`
unchanged.

## Verify (mandatory — simplify edited the tree)

Detect and run the build/test commands using the **Command detection**
table in `pr-loop-lib/steps/04.5-local-verify.md` (first-hit-wins:
CLAUDE.md/AGENTS.md `Build & Test` section, `*.sln`, `package.json`,
`Cargo.toml`, `go.mod`, `Makefile`, `justfile`, `Taskfile.yml`,
`pyproject.toml`/`setup.py`, else skip), under that step's **Execution
rules** (sequential, foreground, timeout ≥ 300000ms, capture
stdout+stderr).

**Do NOT invoke 04.5 as a step.** 04.5 keys both its skip/run decision
*and* its rollback on `context.files_changed_this_iteration`, which is
empty at preflight (iteration 0) — so calling it would silently no-op
the build/test run. Worse, 04.5's second-failure path does a blanket
`git checkout -- <files_changed_this_iteration>`, which on the fold path
would destroy the user's uncommitted work. This step therefore runs the
same commands directly and applies the **path-specific** failure
handling below, whose rollback safety differs between the two paths. (Do
not set `context.files_changed_this_iteration` here either — it is the
comment loop's iteration-scoped field and must stay empty pre-loop.)

## Commit / fold (path-dependent)

### Separate-commit path (committed case)

1. **Secret scan (BLOCKING)** — apply
   `pr-loop-lib/references/secret-scan-rules.md` to `SIMPLIFY_FILES`.
   Any match → halt with the reference's BLOCK message. (`/simplify`
   refactors existing code and is unlikely to introduce a secret, but
   the skill's "secret scan before every commit" rule is unconditional.)
2. **Main-branch guard** — step 01 already HALTs `pr-autopilot` on
   `main`/`master`, so HEAD is a feature branch here. Re-assert
   defensively per `pr-loop-lib/steps/06-commit-push.md`'s guard; halt
   if HEAD is `main`/`master`.
3. **Verify outcome:**
   - Pass → stage exactly `SIMPLIFY_FILES` (never `git add .`/`-A`) and
     commit separately. Subject uses the convention detected in
     `pr-autopilot/steps/04-open-pr.md` §4b (CONTRIBUTING.md, else last
     20 base-branch commits); default `refactor: simplify before review`.
     Body: one line noting this is an automated quality pass. Signing
     follows local git config — never `--no-verify` / `--no-gpg-sign` /
     `-c commit.gpgsign=false`. Emit the `git_commit_argv` audit event
     per step 06's S06.3 procedure.
   - First failure → dispatch one reconcile fixer (same mechanics as
     04.5's first-failure retry, failure body = build/test output),
     re-verify.
   - Second failure → `git checkout -- $SIMPLIFY_FILES` (SAFE here —
     only simplify's edits are reverted; no user work is touched), set
     `context.simplify_outcome = "rolled-back"`, log
     `simplify_rolled_back {reason: "verify-failed-twice"}`, proceed
     to step 02 with `context.head_sha` unchanged. An optional cleanup
     that breaks the build is dropped, not allowed to block the PR.
4. **Advance head:** after a successful commit, set
   `context.head_sha = $(git rev-parse HEAD)`,
   `context.simplify_outcome = "committed"`, and
   `context.simplify_files = $SIMPLIFY_FILES` (the committed paths — for
   the report; step 04 will NOT re-stage them, they are already in
   history). `context.base_sha` is unchanged. Step 02 now reviews the
   simplified diff. Log `simplify_committed {sha: <new head>, files: [...]}`.

### Fold path (overlap case)

Simplify's edits cannot be cleanly separated from the user's uncommitted
work, so they are **not committed here** — step 04 commits them with the
related work as the PR's first commit.

1. **Verify outcome:**
   - Pass → leave the edits in the working tree. **Record `SIMPLIFY_FILES`
     (the `comm -13` set only — files simplify *newly* dirtied, i.e.
     previously-committed/clean PR-changed files) in `context.simplify_files`
     and add them to `context.uncommitted`.** This is required:
     `context.uncommitted` was captured at step 01 *before* simplify ran,
     so a simplify edit to a previously-committed file is dirty but absent
     from that set — without this merge, step 04a's secret scan would skip
     it and step 04b would not stage it, silently dropping the cleanup
     from the PR. Step 04a/04b stage `context.simplify_files`
     **unconditionally** (see `steps/04-open-pr.md` §4a/§4b), which is safe
     precisely because `SIMPLIFY_FILES ⊆ PR-changed code ⊆ related` by
     construction.

     **Do NOT add `OVERLAP_FILES` here.** `OVERLAP_FILES` (the `comm -12`
     set) is files already dirty *before* simplify ran — the user's
     pre-existing uncommitted work. The *related* ones are already in
     `context.uncommitted` and step 04 stages them normally; the
     *unrelated* ones (e.g. a stray `scratch.sh`) must NOT be staged.
     `comm -12` cannot tell the two apart, so including it would stage
     unrelated dirty files and violate this step's own guarantee (see the
     "Path selection" note above) that unrelated dirty files are never
     staged by this step. Overlap files therefore need no handling here:
     they ride step 04's normal `context.uncommitted` path, and simplify's
     extra edits to a *related* overlap file ride along with it.

     Set `context.simplify_outcome = "folded"`. Log `simplify_folded
     {files: [...], reason: "intermingled-with-uncommitted-work"}`.
     `context.head_sha` unchanged (nothing committed).
   - First failure → dispatch one reconcile fixer, re-verify.
   - Second failure → **do NOT auto-`git checkout`** — that would
     destroy the user's uncommitted implementation work in the
     overlapping files. Log an `error {stage: "01.5-simplify",
     error_type: "verify-failed-twice-fold", message: <build/test
     failure summary>}` event (this is the `error` event the
     `simplify_rolled_back` taxonomy note points to for the fold path),
     then HALT with:

     ```
     HALT — /simplify edits combined with your uncommitted changes
     break the local build/tests and could not be auto-reconciled.

     Your working tree contains BOTH your uncommitted work and the
     simplify edits in: <OVERLAP_FILES>

     Review `git diff`, fix or selectively revert, then re-invoke
     pr-autopilot. (No files were reverted automatically, to avoid
     losing your work.)
     ```

     Do not open the PR.

## Dry-run

When `--dry-run` is set: run `Skill(skill="simplify")` and the local
verify (both reversible, local-only), but do **not** commit. Write the
intended commit message and `SIMPLIFY_FILES` list to the dry-run dir
(`$DRYRUN_DIR/simplify-commit.txt`) and print its path. Leave the
simplify edits in the working tree and proceed to step 02. This matches
the SKILL.md dry-run rule (skip `git`/`gh`/`az` mutations; preserve their
inputs as files).

Context fields are not git/gh/az mutations, so the fields describing
staging intent are still written under `--dry-run` to keep the dry-run
report accurate: on the **fold path**, still record `context.simplify_files
= SIMPLIFY_FILES`, merge it into `context.uncommitted`, and set
`context.simplify_outcome = "folded"`; on the **separate-commit path**, set
`context.simplify_files = SIMPLIFY_FILES` but leave `context.head_sha`
unadvanced and `context.simplify_outcome` unset-by-commit (no commit
occurred) — the `simplify-commit.txt` file is the record of what would
have been committed.

## Output (context fields)

- `context.head_sha` — advanced to the simplify commit on the
  separate-commit path; unchanged otherwise.
- `context.simplify_outcome` — one of `committed`, `folded`,
  `no-changes`, `rolled-back`, `skipped`. Carried for the final report.
- `context.simplify_files` — the touched-file list. Populated on the
  `committed` and `folded` outcomes; left `[]` on `skipped`,
  `no-changes`, and `rolled-back`. Its meaning differs by outcome:
  on `committed` the files are already in history (step 04 does **not**
  re-stage them); only on `folded` does step 04 stage them (§4a/§4b,
  unconditionally). For the report in both cases.

## Invariants

- If `context.simplify_outcome == "committed"`, `context.head_sha` differs
  from its step-01 value and `git cat-file -t <head_sha>` is `commit`.
- No `git checkout` was issued in the fold path (overlap case never
  auto-reverts).
- The simplify commit (if any) used no signing-bypass / hook-skipping
  flags — same S06.3 predicate as the loop's step 06.
